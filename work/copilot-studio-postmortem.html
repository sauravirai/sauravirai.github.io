TL;DR

The plan we wrote ≠ the plan we shipped — and that's the lesson.

We started with the “safe” enterprise architecture: SharePoint list + Power Automate HTTP-trigger flow for user preferences and the room catalog. After actually building toward it, we ripped it out in a single commit and replaced it with OneDrive Graph API + an inline room catalog in the agent's instructions — zero custom backend, zero flow, zero new lists.

Two real bugs blocked publish: a one-letter typo in an OpenAPI operationId, and a DLP policy block that we cleared by routing through the Teams Admin Center channel rather than publishing direct. The Microsoft 365 /places endpoint quietly 403s in our tenant, so room discovery is hand-curated instead.

Result: a working Copilot Studio agent, live in Teams + Outlook compose, with no Power Automate, no SharePoint, no custom backend, and a scaffold someone else can ship from in under an hour.

Attempt #1 — The “safe enterprise” plan

SharePoint UserPreferences list · RoomCatalog list · Power Automate HTTP-trigger flow

📋 SharePoint + Power Automate as the prefs backend

Abandoned

What we planned to build

  • A SharePoint Team site (MeetingRoomAssistant) with two lists: UserPreferences (~21 columns, lots of *Json multi-line fields) and RoomCatalog (room emails, capacity, tier).
  • A single Power Automate HTTP-trigger flow doing switch-on-action routing for getPreferences / updatePreferences / getRooms.
  • A custom OpenAPI plugin (preferences-openapi.yaml) pointing at the flow's POST URL, surfaced in Copilot Studio as preferences_oauth_ref.
  • The full design lives in sharepoint-setup.md in the repo if anyone wants to recreate it.

Why it looked like the right answer

  • It's the “official Microsoft pattern” for stateful Copilot Studio agents — the docs actively recommend it.
  • SharePoint is already governed, backed up, and audit-friendly.
  • RoomCatalog-as-a-list meant a new building = a new row, no code change. Genuinely a nice property.

What pushed us off it

  • Dependency surface area. SharePoint site + 2 lists + Power Automate flow + custom OpenAPI + custom OAuth wiring + 21 column definitions to keep in sync with the JSON schema. Every link in that chain needed governance, monitoring, and a re-auth dance every ~90 days.
  • The flow was its own little app. A Switch with Get item → Parse JSON → Update item per branch. Debugging meant tabbing into make.powerautomate.com, finding the run, expanding nodes — not a great inner loop.
  • No genuine multi-user benefit. Every user gets their own preferences anyway. We were paying the “shared list” tax for a use case that's actually per-user data.
  • OneDrive already exists for every M365 user. Get file content + Create file against /Documents/meeting-room-agent-prefs/preferences.json uses the user's own OneDrive — no shared resource, no auth gymnastics, no flow to monitor.
⚠️ Cost of this detour A full day of scaffolding (sharepoint-setup.md, preferences-openapi.yaml, preferences-plugin.json) before the simpler path became obvious. The files are still in the repo as a cautionary reference — kept on purpose, not deleted.

The pivot — “wait, OneDrive already does this”

Commit 05fa355 · 19 Apr 2026 · the smallest commit that fixed the most

🌀 Drop SharePoint + Power Automate — prefs via OneDrive Graph API

Single-commit pivot

One commit deleted the entire custom prefs stack. Three things happened in the same change:

❌ What we deleted
  • SharePoint site provisioning steps
  • 2 list schemas (UserPreferences, RoomCatalog)
  • Power Automate HTTP-trigger flow
  • Custom preferences_oauth_ref connection
  • preferences-plugin.json as a separate plugin action
  • The whole “Phase 1 — SharePoint setup” in the deploy plan
✅ What replaced it
  • OneDrive Graph API — Get file content + Create file against the user's own OneDrive at /Documents/meeting-room-agent-prefs/preferences.json
  • Built-in OAuth — no custom connection to provision
  • Room catalog inlined as a markdown table inside instructions.md — agent reads it directly
  • Silent fall-through to onboarding flow if the file doesn't exist (404 = new user, not an error)
✅ Net effect Zero custom backend. Zero flow to monitor. Zero shared resource. Each user owns their own preference file in their own OneDrive — better privacy posture, simpler auth, smaller blast radius. The room catalog refresh is now “edit a markdown table and republish the agent” — same number of steps as a SharePoint list edit, but no list to govern.

The trade-off we accepted

  • Room catalog updates = re-publish. Adding a new floor means editing instructions.md and pushing a new agent version. The SharePoint approach would have allowed live edits. We decided this is fine because new floors happen monthly at most, not daily.
  • Cross-user analytics are harder. Each user's prefs live in their own OneDrive — can't run a SQL-style query across all users. Acceptable for a productivity tool; would not be acceptable for, say, a compliance system.

Bugs & gotchas we shipped through

The kind of thing the docs don't warn you about.
🐛 The one-letter typo that silently broke getEvent
Severity: blocker Fixed in 3b4736a graph-openapi.yaml
Symptom The agent could list calendar events fine, but every call to fetch a single event came back with no payload. No clear error in the Copilot Studio test pane. The action just… returned nothing.
Root cause In graph-openapi.yaml, the operation was declared with operatid: getEvent instead of operationId: getEvent. Copilot Studio silently dropped the action because the required field was missing — but didn't surface a validation error during package import.
Fix Rename the field to operationId. Re-zip. Re-upload. Test passes.
Lesson Run swagger-cli validate graph-openapi.yaml before every package re-zip. Copilot Studio is forgiving on import but ruthless at runtime.
🛡️ DLP policy hard-blocked the direct publish path
Severity: blocker Worked around in 407a829 Org policy
Symptom Publishing the agent directly from Copilot Studio → Channels → Microsoft 365 Copilot was blocked by the Walmart Data Loss Prevention policy. The agent worked perfectly in the Copilot Studio test pane but never surfaced inside Teams or Outlook compose.
Root cause (working theory) The org DLP policy treats “direct publish to M365 Copilot” differently from “Teams app sideload & publish via Teams Admin Center.” The TAC path goes through the Walmart-managed app catalog, which is already trust-tier-approved. The direct publish path bypasses that trust tier and triggers a policy denial. (If anyone from the M365 admin team can confirm or correct this, please update this doc.)
Fix Don't publish via “Channels → Microsoft 365 Copilot → Publish.” Instead:
  1. Zip the agent package (manifest + agent JSON + plugin files + icons).
  2. Sideload in Teams via Apps → Manage your apps → Upload a custom app for personal testing.
  3. Upload to Teams Admin Center → Manage apps → Upload and publish with a permission policy scoped to your team.
  4. The agent appears in Teams + Outlook compose for everyone in the policy scope.
Lesson At Walmart, treat the Teams Admin Center as the only real publish path for Copilot agents. The “publish from Copilot Studio” UI is for tenants without strict DLP — not us.
🚪 Microsoft Graph /places returns 403 in our tenant
Severity: workaround in place Documented in instructions.md
Symptom The natural way to discover bookable rooms is GET /me/findRooms or GET /places/microsoft.graph.room. Both return 403 Forbidden in our tenant.
Working theory Walmart's Graph permissions don't grant Place.Read.All at app or delegated scope by default. Requesting it would require an admin-consent flow we chose not to fight.
Workaround Hand-curated room catalog inside instructions.md — populated from Outlook → New Meeting → Add a room (which uses a different, allowed Graph code path). New buildings = edit the markdown table and re-publish.
Trade-off Catalog drift. If facilities adds a new room and nobody updates the agent, it won't be bookable. Acceptable for ~7 rooms; would not scale to ~700.
🔄 Preference drift between the AgentRunner CLI and the Copilot Studio agents
Severity: ongoing Two-system trade-off
Symptom The AgentRunner terminal agent stores preferences in a local YAML file. The Copilot Studio agent stores them in OneDrive. Edit one, the other doesn't know.
Mitigation today The AgentRunner agent was updated to show a sync reminder every time prefs change — it tells you exactly which fields changed and links to the OneDrive file. The reverse direction (Copilot Studio → CLI) is unmitigated; we just treat OneDrive as the source of truth and accept that the CLI may be slightly stale.
Open question Should the AgentRunner agent read from OneDrive directly via msgraph_* tools instead of keeping a local YAML? Adds latency but eliminates drift. Tracked as a roadmap item.

Things we learned (TIL)

If you're about to build a Copilot Studio agent at Walmart, read this first.
TIL #1
Reach for OneDrive before SharePoint

Per-user JSON in OneDrive is the simplest, most-private place to store agent preferences. SharePoint is for genuinely shared state — not user profiles dressed up as a list.

TIL #2
Validate every OpenAPI before re-zipping

Copilot Studio import is forgiving; runtime is not. A typo in operationId will silently drop the action with no error. Add swagger-cli validate to your loop.

TIL #3
Always publish via Teams Admin Center

The “direct publish to M365 Copilot” path is DLP-blocked at Walmart. TAC is the only reliable route to your users. Plan for the policy review queue.

TIL #4
Inline small reference data in the prompt

For ~10s of items (rooms, locations, escalation contacts), a markdown table inside instructions.md beats a list-backed plugin. Republish-to-update is fine at that scale.

TIL #5
Built-in connectors > custom plugins, every time

The OneDrive and Outlook built-in connectors handle OAuth, refresh, and rate limits for you. Reach for a custom OpenAPI plugin only when no built-in covers your need.

TIL #6
Silent fallback > error message for new users

If Get file content 404s for a new user, never show the connector error. Fall through to the onboarding flow as if it was always the plan. Connector errors confuse non-technical users.

TIL #7
Two agents (CLI + Copilot Studio) is fine

Don't try to make one agent serve both power-users and EAs. The CLI is your debug surface; Copilot Studio is the user surface. Keep them deliberately separate; pick one source of truth (OneDrive).

TIL #8
Keep the abandoned scaffolding in the repo

sharepoint-setup.md, preferences-plugin.json, and preferences-openapi.yaml are still in the repo as a cautionary reference. Useful for future folks evaluating the same trade-off.
