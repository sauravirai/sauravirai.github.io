📅 meeting-room-agent 🟢 PORT NOW

Calendar & meeting room assistant for Microsoft 365.

Why it's begging to be on InternalAgentPlatform

  • Replace local preference store with Cosmos DB (InternalAgentPlatform-native).
  • Wrap existing tool surface in A2A endpoints.
  • Auth: dual auth (OAuth2 + SR headers) — works because msgraph is already user-context-driven.
  • Retire the Copilot Studio version; one InternalAgentPlatform agent with multiple front-ends.

📖 storyteller 🟡 PORT W/ UX REWORK

Turns a Walmart associate's AgentRunner journey into a polished HTML slide deck.

The trade-off

✅ Appeal: Universal use case. Every Walmart associate could use this.

⚠️ Tricky: Heavy reliance on ask_user_question for the 3-round interview. A2A supports streaming + push notifications, so it CAN handle conversational flows — but the current prompt assumes a single interactive session. You'd need to redesign as either:

  • Multi-turn A2A conversation (Message payloads + session state in Cosmos)
  • Web form-based intake → generate deck

Verdict: Treat as a rewrite, not a lift-and-shift. Output (self-contained HTML deck via share-puppy) is already centralized.

🪢 daily-task-agent 🟡 TEMPLATE ONLY

Daily todo aggregator + nudger across Outlook/Jira/Confluence/OneNote/Slack.

Why the pattern is great, but YOUR instance shouldn't port

✅ Pattern value: "Aggregate scattered todos + draft nudges" is a killer enterprise capability.

⚠️ Risks of porting your instance as-is:

  • Sends emails/Slack on your behalf — from a centralized hosted agent, identity propagation must be airtight. Blast radius = real (a "from Sauravi" mail without your approval).
  • Invokes Sauravi-org-context-agent — which won't exist on InternalAgentPlatform.
  • Writes to ~/todos/ and reads ~/.config/slack_webhook — both local.

Recommendation: Build a generic "Daily Digest Agent" on InternalAgentPlatform for any Director, parameterized by Jira filters, OAuth tokens (not webhooks), and user-configurable nudge templates. Keep YOUR personalized instance on AgentRunner.

🏗️ Sauravi-org-context-agent 🔴 KEEP LOCAL

Private specialist for re-org transitions, stakeholder maps, calibration.

Why this should never be on InternalAgentPlatform
  • Ground truth file Sauravi-org-ground-truth.md contains: direct reports, HRBP, comp/calibration context, MIP/equity guidance, reorg history with departed employees by name and date.
  • That's PII-grade HR/comp data. Even with InternalAgentPlatform's centralized governance, the right place is a private file on your laptop.
  • Agent is explicitly designed to refuse third-party data sharing (per daily-task-agent notes). On a shared platform that contract is harder to enforce.
  • Walmart rule: no HIPAA on InternalAgentPlatform; this isn't HIPAA but is the same risk class for HR data.

Verdict: Textbook "right tool, right place" — local AgentRunner agent with file-based ground truth is exactly the correct architecture.

🐑 stack-shepherd 🔴 KEEP LOCAL

Tends Sauravi's published agent stack at ~/workspace/Sauravi-agent-stack/.

Why it shouldn't be ported
  • Job is to tend YOUR repo: runs git push, executes nudge.sh, mutates local files.
  • A hosted InternalAgentPlatform agent has no access to your laptop's filesystem or SSH keys.
  • Multi-team value is essentially zero — portfolio steward for one person.
  • The whole point is local file orchestration. Putting it on a server is the wrong primitive.

Verdict: Textbook local CLI orchestration agent. InternalAgentPlatform would actively break its core capability.
